Linux IO 协议栈图示

By Lirenke

Published Dec 28 2014

Contents

在网上找到了个好东东，表示Linux当前版本IO路径的图例，以老朽现在的功力不能全部看明白。

个人旁白：
最上层是应用程序，也可以理解成GuestOS，因为后续看到了（virtio）。
APP通过os的api调用访问存储，这里描绘了三种方式：

  1. directIO
  2. VFS
  3. page cache

directIO即是说，带上O_DIRECT标志位后，APP会绕开文件系统和页缓存，直接访问块存储，按图上描绘精确点说应该是Block I/O层面。这个标志位的作用是在Linux下不会产生大量的cache导致free内存不足；另外，以前在搞FUSE实现ROFS时，有些节点的内容是动态变更的，这个标志位用来绕开页缓存从而获取实时的数据。

Page cache，页缓存，载体就是物理内存。这种形式是OS在I/O设计上的一种提高访问效率的通用手段。OS会把当前经常访问的或者是最近访问的数据保存到这里，下次APP在来访问同一处数据时，就可以直接从页缓存中获取，不必在经过文件系统再到底层硬件。在Linux上，用“free”查看，cache字段即表示当前在页缓存上的数据大小，默认是持续上升的，所以经常会发现Linux的free内存大小不足，这需要用户手动的去清理cache空间。另外，openstack的nova在判断物理机可用内存大小时，是会扣除cache大小的。

VFS，这个是Linux最典型通用的形式。OS为了支持多套不同形式的文件系统类型，提供了一套通用的规范接口，对APP而言，它们只需调用VFS的接口即可，不用关心真正的文件系统类型。VFS层面会根据调用的内容去调用对应的，真正的文件系统接口。所以这张图中，看到了如此众多的文件系统类别。

pseudo FS和special purpose FS两者不是很熟悉，前者叫伪文件系统，以前我的理解就是内存文件系统，比如/proc，但和后者的区别不是很了解。

再往下层，就是Block I/O层，我只认识device mapper和lvm，这层位于FS之下，物理设备之上，应该还是属于OS定义的逻辑层面。比如lvm的逻辑卷。通过I/O调度映射到了各种驱动层上去了。

下面就是各种驱动层级了，要补充的是最左边有个”request-based device mapper targets”，看到dm-multipath，就能猜到就是在SCSI协议下的多路径模块了。图上把SCSI的路径信息描绘的很细，分成了三层，SCSI上层，SCSI中层和下层。上层即是linux暴露的块设备文件，如/dev/sda，中层没展开，下层就是真正的SCSI协议驱动层了，我认识的就libata，这玩意把原来并行ATA的IDE设备名从”hd”统一到了”sd”。ahci,这个也眼熟，在os的bios下可以设置磁盘的访问协议，sata硬盘下必须选ahci，否则蓝屏。。。

最下面就是物理设备，就是硬盘了，有HDD，SSD，DVD，LSI的RAID，其他的如HBA应该是走FC协议的适配器。

图上的其他部分，如sysfs，/dev/fio等等不是很了解。
